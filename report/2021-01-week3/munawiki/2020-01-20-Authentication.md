**암호화와 hashing, salting 등의 개념을 이해할 수 있다.**

- 암호화는 양방향 ==> 복호화가 가능 
- hashing은 단방향 ==> 같은 문자열에 대해서는 항상 같은 다이제스트가 나오기 때문에 레인보우 테이블에 대한 취약점이 있음
- salting은 hashing에서 항상 같은 다이제스트가 나오는 취약점을 보완하기 위해 임의의 문자열을 추가해서 같은 문자열에 대해 다른 다이제스트가 나오도록 함


**HTTP와 HTTPS의 차이점을 이해할 수 있다.**

- http: 데이터 암호화가 구현되지 않음 **중간자 공격**에 취약함
- https: 암호화된 연결 


**권한 부여(Authorization)와 인증(Authentication)에 대해 이해할 수 있다.**
- 권한 부여(Authorization): 사용자에게 특정 리소스 또는 기능에 대한 접근 권한을 부여하는 프로세스
- 인증(Authentication): 사용자가 누구인지 확인하는 행위 비밀번호, 일회성 핀, 생체 인식 등이 여기에 포함 됨


**어떻게 브라우저에 쿠키가 저장되는지 이해할 수 있다.**
 - 서버에서 클라이언트로부터 요청을 받고 응답 객체에 쿠키를 넣어 줌

브라우저에서 저장된 쿠키를 확인하는 법을 알 수 있다.
- 크롬 개발자 도구 -> application -> storage -> cookies 에서 확인 가능


**쿠키가 요청에 어떻게 전송되는지 이해할 수 있다.**
- 헤더에 담겨서 전송이 되었음


**전송된 쿠키를 서버에서 어떻게 조회하는지 알 수 있다.**
- request 객체의 헤더에 있었음

**쿠키의 한계를 이해할 수 있다.**
- 쿠키 데이터의 암호와 여부에따라 해커가 쿠키 데이터를 읽거나 액세스 권한을 얻을 수 있음


**인증 외에 어떤 목적으로 쿠키가 사용되는지 알 수 있다.**
- 사용자의 입력 정보를 기억하고 다시 입력할 필요가 없게 함
- 쿠키에서 수집한 데이터로 제품 그룹, 지리적 위치, 검색어 및 인구통계에 대한 구체적인 마케팅에 활용 가능

**세션의 개념을 이해할 수 있다.**
- HTTP는 stateless이기 때문에 다른 요청으로 연결하기 위해서는 요청 사이에 사용자 데이터를 저장하는 방법이 필요


**쿠키와 세션은 서로 어떤 관계이며, 각각이 인증에 있어서 어떤 목적으로 존재하는지 이해할 수 있다.**
- 쿠키: 클라이언트에 저장
- 세션: 서버에 저장


**세션의 한계를 이해할 수 있다.**
- 서버 메모리에 저장된 세션 데이터로 인해 사용자가 많은 경우 오버헤드
